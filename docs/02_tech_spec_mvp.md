# Техническое задание на MVP

## 1. Общие требования
Нужно разработать MVP системы доверенных сообщений для Aerotrust Technics:
- Telegram-бот для отправителей
- backend API
- PostgreSQL
- файловое хранение вложений
- административная веб-панель
- базовая аналитика

## 2. Основные сценарии пользователя

### 2.1 Первый вход
1. Пользователь открывает Telegram-бота.
2. Бот показывает приветствие.
3. Бот запрашивает invite code.
4. Если код валиден — пользователь авторизуется и получает доступ к меню.
5. Если код невалиден — бот просит повторить ввод.

### 2.2 Создание сообщения
1. Пользователь нажимает «Отправить сообщение».
2. Выбирает режим:
   - Анонимно
   - Открыто
3. Выбирает категорию:
   - Безопасность
   - Качество
   - Процессы
   - Поведение / этика
   - Улучшения
   - Другое
4. Вводит текст сообщения.
5. По желанию прикладывает фото или файл.
6. Подтверждает отправку.
7. Бот отвечает: «Сообщение принято».

### 2.3 FAQ / правила
Бот должен уметь показывать:
- как работает система;
- что значит анонимный формат;
- какие сообщения можно отправлять.

## 3. Функциональные требования к Telegram-боту
- поддержка onboarding;
- проверка invite code;
- сохранение состояния пользователя;
- сценарий создания сообщения шаг за шагом;
- поддержка текста;
- поддержка media/file attachments;
- главное меню;
- FAQ / помощь;
- подтверждение отправки;
- защита от спама (rate limiting на базовом уровне).

## 4. Требования к анонимности
### 4.1 Анонимный режим
В административной панели для анонимных сообщений не отображать:
- username Telegram;
- имя пользователя из Telegram;
- Telegram ID;
- иные явные идентификаторы.

### 4.2 Открытый режим
В административной панели отображать:
- display name (если возможно сформировать);
- технический идентификатор пользователя;
- признак, что сообщение отправлено открыто.

## 5. Административная веб-панель
Нужна простая web admin panel с авторизацией.

### 5.1 Функции
- вход по логину/паролю;
- просмотр списка сообщений;
- фильтр по:
  - категории,
  - зоне,
  - статусу,
  - периоду,
  - режиму (анонимно / открыто);
- открытие карточки сообщения;
- просмотр вложений;
- изменение статуса;
- назначение зоны;
- таблица invite codes;
- аналитический экран.

### 5.2 Роли
MVP:
- `admin`
- `resolver`

### 5.3 Зоны
Сообщения должны относиться к зонам обработки. На MVP зона может совпадать с категорией либо задаваться mapping-таблицей:
- Безопасность → safety
- Качество → quality
- Процессы → process
- Поведение / этика → ethics
- Улучшения → improvements
- Другое → general

## 6. Аналитика (MVP)
Показать:
- количество сообщений за период;
- по категориям;
- по зонам;
- по статусам;
- доля анонимных / открытых;
- динамика по дням / неделям;
- среднее время до закрытия.

## 7. Нефункциональные требования
- backend должен быть container-ready;
- конфигурация через env variables;
- логирование без утечки чувствительных данных;
- валидация входных данных;
- базовая обработка ошибок;
- код должен быть структурирован и пригоден для доработки;
- README по запуску обязателен.

## 8. Рекомендуемый MVP-стек
- Python 3.12
- FastAPI
- aiogram 3
- PostgreSQL
- SQLAlchemy 2 + Alembic
- Redis (опционально, но желательно для FSM/rate-limit/cache)
- React / Next.js или простая серверная админка
- Docker Compose
- Nginx
- Object/file storage на локальном диске сервера на MVP

## 11. Критерии приемки MVP
MVP считается готовым, если:
1. Бот запускается и принимает invite code.
2. Пользователь может отправить анонимное и открытое сообщение.
3. Можно приложить вложение.
4. Сообщение сохраняется в БД.
5. Сообщение видно в админ-панели.
6. Для анонимного сообщения в админке не видны данные отправителя.
7. Можно менять статус сообщения.
8. Аналитический экран показывает базовые метрики.
9. Есть docker-compose и инструкция запуска.

# Отчёт о технической проверке

Дата: 2026-07-16  
Объект: репозиторий системы доверительных сообщений (Telegram-бот, backend, admin, Docker Compose).  
Цель: подтверждение готовности к промышленному развёртыванию на сервере принимающей стороны.

## 1. Объём проверки

Проверены:

1. Структура проекта и состав сервисов
2. Telegram-бот: чтение токена из переменных окружения, обработчики, вложения, обработка ошибок, rate limit
3. Backend API: `/healthz`, авторизация, RBAC, обработка ошибок, отключение OpenAPI при `APP_ENV=prod`
4. Админ-панель: подключение к API, работа при пустых данных, разграничение доступа на уровне API
5. База данных PostgreSQL, миграции Alembic, Docker volumes
6. Docker Compose (среда разработки и production)
7. Сохранность файловых вложений и безопасность путей
8. Конфигурация секретных данных и `.gitignore`
9. Безопасность публикации портов и анонимность на уровне API/UI
10. Скрипты развёртывания и резервного копирования
11. Базовая проверка запуска и сохранности данных после перезапуска

## 2. Архитектура

```
Telegram ←polling→ bot ──┬──► PostgreSQL
                         ├──► Redis
                         └──► volume uploads
Браузер → admin (Nginx) → backend (FastAPI) → PostgreSQL + uploads
```

## 5. База данных и сохранность данных

- Используется PostgreSQL 16 в Docker
- Данные размещаются в named volume `postgres_data` / `aerotrust_postgres_data`
- Вложения размещаются в named volume `uploads_data` / `aerotrust_uploads_data`
- Миграции выполняются через Alembic (`alembic upgrade head`)
- Команда `docker compose down` (без `-v`) сохраняет данные
- Команда `docker compose down -v` удаляет данные — запрещена в штатной эксплуатации без backup

## 6. Секретные данные

- Реальный `.env` исключён из git
- Шаблоны `.env.example` и `.env.production.example` содержат плейсхолдеры
- Описание переменных и секретов вынесено в `ENVIRONMENT_VARIABLES.md` и `SECRETS_REQUIRED.md`

## 7. Анонимность

- Для режима `anonymous` API и UI админки не возвращают данные автора
- В базе данных поле `author_user_id` может сохраняться для сценария просмотра собственных обращений пользователем в боте
- При прямом доступе к БД возможна корреляция автора; это эксплуатационное ограничение MVP

## 8. Оставшиеся эксплуатационные замечания

1. Автоматические ночные backup требуют настройки на стороне сервера
2. Анонимность реализована на уровне API/UI, не как криптографическая несвязность записей
3. Конфигурация рассчитана на одиночный хост Docker без высокой доступности
4. Работа бота зависит от исходящего доступа к Telegram API
5. HTTPS не встроен в compose и настраивается reverse proxy на хосте

## 9. Рекомендуется проверить вручную на production-сервере

- Полный сценарий подачи обращения через Telegram
- Скачивание вложений из админки
- Корректность TLS и DNS
- Расписание и восстановление из backup
- Политика хранения и ротации секретных данных
- Integration-тесты backend при наличии тестовой БД

## 10. Документация комплекта

Актуальный комплект технической документации находится в каталоге `deploy_bundle/` и включает инструкции по развёртыванию, требованиям к серверу, переменным окружения, секретам, резервному копированию, обновлению и чеклисту перед запуском.

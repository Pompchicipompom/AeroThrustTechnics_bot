# Чеклист перед запуском в production

## Конфигурация

- [ ] Файл `.env` создан из `.env.production.example`
- [ ] Заполнены `TELEGRAM_BOT_TOKEN`, `POSTGRES_PASSWORD`, `DATABASE_URL`, `ADMIN_JWT_SECRET`
- [ ] Пароль в `DATABASE_URL` совпадает с `POSTGRES_PASSWORD`
- [ ] `APP_ENV=prod`
- [ ] Реальные секреты не находятся в git

## Инфраструктура и данные

- [ ] Установлены Docker и Docker Compose
- [ ] PostgreSQL использует persistent volume
- [ ] Uploads используют persistent volume
- [ ] PostgreSQL и Redis не опубликованы наружу
- [ ] Открыты только необходимые порты (80/443 или корпоративная схема доступа)
- [ ] Настроено резервное копирование (см. `BACKUP_AND_RESTORE.md`)

## Первый запуск

- [ ] Выполнен `bash infra/scripts/deploy_vps.sh` (или эквивалентная последовательность)
- [ ] Миграции применены (`alembic upgrade head`)
- [ ] `curl /healthz` возвращает `status: ok`
- [ ] Доступна страница `/admin/`
- [ ] Создана учётная запись администратора
- [ ] Создан invite-код
- [ ] HTTPS настроен (рекомендуется)

## Функциональная проверка

- [ ] Админка требует авторизацию
- [ ] Бот отвечает на `/start`
- [ ] Тестовое обращение сохраняется
- [ ] Обращение отображается в админке
- [ ] Смена статуса обращения работает
- [ ] Вложение (если используется) сохраняется и доступно для скачивания
- [ ] После `docker compose ... restart` данные сохраняются
- [ ] При `APP_ENV=prod` недоступен `/docs`

## Анонимность

- [ ] Для анонимного обращения в API/UI админки автор не отображается
- [ ] Для режима «открыто» отображение автора соответствует продуктовым правилам

## Эксплуатация

- [ ] Известен запрет на `docker compose down -v` без backup
- [ ] Документированы команды просмотра логов и обновления
- [ ] Назначены ответственные за хранение секретов и backup
